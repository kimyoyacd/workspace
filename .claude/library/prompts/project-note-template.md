# 프롬프트 원형 — 프로젝트 노트 템플릿

> `.claude/projects/YYYYMM_프로젝트명.md` 신규 생성 시 이 구조를 그대로 복사해서 쓴다.
> 목적: 세션이 끝나도 "결정사항"은 요약으로 바로 보이고, 그 판단의 근거가 된 원본 대화·자료는 유실 없이 append로 쌓인다.

```markdown
# (프로젝트명)

- **클라이언트**: (미정/확정)
- **업무 범위**: (한 줄)
- **일정**: (착수~납기)
- **견적**: (금액/상태, 확정 전이면 "협의중")
- **상태**: (현재 단계 한 줄)

## 요약 (매 세션 종료 시 갱신)
> 이 섹션은 항상 "지금 기준 최신 상태"만 남긴다. 과거 버전은 여기서 지우고 아래 원본 로그에 맡긴다.

- **핵심 결정사항**: (무엇을 결정했나, 결론만)
- **결정 근거**: (왜 그렇게 했나, 한 줄~두 줄. 상세 논의는 원본 로그 참조)
- **다음 액션**: (누가/무엇을/언제까지)

## 원본 로그 (append-only, 절대 덮어쓰지 않음)
> 세션에서 오간 논의·시행착오·버려진 대안·클라이언트 반응 등 디테일을 날짜별로 그대로 누적한다.
> 나중에 "왜 그렇게 했더라?"를 복원해야 할 때만 펼쳐본다. 요약에서 뺐다고 삭제하지 않는다.

### YYYY-MM-DD
- (그날 세션에서 오간 내용, 결정과 근거, 버린 대안과 이유를 시간순으로)

### YYYY-MM-DD
- ...
```

## 사용 규칙
1. **요약 섹션은 덮어쓴다.** 최신 결정 상태만 담아 실장님이 위에서 훑으면 바로 파악되게 한다.
2. **원본 로그 섹션은 무조건 append한다.** 세션 마무리 시점(또는 SessionEnd 훅)에 그날 대화·결정 과정을 날짜 헤더로 추가만 하고, 기존 로그는 절대 고치거나 지우지 않는다.
3. 파일이 길어지는 건 정상이다. 필요한 부분은 Grep으로 찾아 읽으면 되므로 로그를 줄이려고 요약하지 않는다.
4. 요약은 "무슨 얘기를 했다"가 아니라 "무엇을 결정했고 왜인지"만 압축한다 — 대화 전 과정을 요약하려 하면 결국 다시 정보가 깎인다.
