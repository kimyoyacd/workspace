# 디자인 프로젝트 범용 덱 와이어프레임 (목차 골격)

> **적용 범위**: 브랜드 제안뿐 아니라 홈페이지·배너·게임 UI·상세페이지 등 **모든 디자인 관련 프로젝트**. 결국 어떤 디자인이든 "하나의 대상을 브랜드처럼 정의하고 새 컨셉을 제안하는" 동일 패턴을 따르므로, 시장조사→발산→전략 정의→비주얼 시스템→최종 제안의 골격은 공유한다.
> `brand-design-proposal.md`(브랜드 전용 상세 파이프라인)의 상위 개념. 브랜드가 아닌 프로젝트는 이 와이어프레임만 따르고, Manus/Runable 세부 게이트는 생략하거나 축약해도 된다.

## 목차 골격 (deck.html 섹션 순서)

```
0. Cover — 프로젝트명 + 한 줄 관점
1. 시장·경쟁 리서치
   1a. 동일 카테고리 경쟁사 N개 (가격대/타깃/핵심메시지/출처 전체 필드)
   1b. 이종업계 확장 리서치 (철학적 공명 기준, 카테고리 편식 방지)
   1c. 클러스터링 + 3열 흐름표 (클러스터→전략→무드)
2. 관점 정의 (Gate 1 상당)
   - 한 문장 관점 + 이 관점이 필요한 이유 + 경쟁사와 다른 믿음 + 타깃 반응 감정
3. 발산 (브레인스토밍)
   - 방향 후보 다수(질보다 양) → 카테고리 묶음 → 확장 추천 2~3선
   - 여기서 그로테스크/과함 등 리스크 있는 방향이면 critic으로 즉시 검증
4. 디자인 전략 (무드보드)
   - 그래픽 모티프·형태 / 타이포·로고타입 / 컬러·질감 — 방향별로 나란히
   - 실제 생성 이미지로 무드슬라이드 완성(텍스트 설명만으로 끝내지 않음)
5. 레퍼런스 연결 & 포지셔닝맵
   - 발산된 각 방향이 어떤 레퍼런스에서 무엇을 계승/배제하는지
   - XY 포지셔닝맵으로 화이트스페이스 시각화
6. 최종 시스템 슬라이드 (Runable 3, `runable-3-slide-templates.md` 6종 재사용)
   - Brand/Project Story, Value Keywords, Mind Map, Direction Spectrum, Graphic Motif, Competitor Research
7. 적용 로드맵 (Application Roadmap)
   - 확정된 방향이 실제 채널/화면별로 어떻게 적용되는지 (웹/배너/UA/상세페이지/게임UI 등 프로젝트 성격에 맞게)
   - "다음 실행 단계" 타임라인
```

## 각 섹션 최소 요건 (체크리스트)

```md
[ ] 0. Cover에 한 줄 관점이 있는가?
[ ] 1a. 경쟁사 카드가 전체 필드(가격/타깃/메시지/출처)를 갖췄는가? 개수 요청분 누락 없는가?
[ ] 1b. 이종업계 리서치를 별도로 했는가, 동일 카테고리에만 갇히지 않았는가?
[ ] 2. 관점 한 문장이 있고 사용자 확인을 받았는가?
[ ] 3. 발산이 다수 방향으로 이뤄졌고, 리스크 있는 방향은 검증했는가?
[ ] 4. 무드보드가 텍스트뿐 아니라 실제 생성 이미지로 채워졌는가?
[ ] 5. 포지셔닝맵이 실제 리서치 데이터 기반인가(임의 배치 아님)?
[ ] 6. Runable 3 슬라이드 6종이 최신 확정 방향과 일치하는가(구버전 잔재 없는가)?
[ ] 7. 로드맵/다음 단계가 명시되어 있는가?
```

## 버전 전환 시
방향이 통째로 바뀌면(`00-project-index.md` 참조) 이 목차 골격 그대로 새 버전 파일에 재적용한다 — 골격은 유지, 내용만 교체. 이전 버전은 지우지 않고 인덱스에 "폐기됨"으로만 표시.
