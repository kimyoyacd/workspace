# Thinking OS — 김효정 평생 사고 운영체계

- **클라이언트**: 없음 (내부 · 김효정 개인 자산 프로젝트)
- **업무 범위**: 콘텐츠 자동생산 아님. 사고방식·철학·문체·디자인 취향을 학습·기록·축적하는 운영체계 설계
- **상태**: 1단계(Identity) 완료 — `202607_thinking-os/Identity.md` 참조. 2단계(Thinking 캡처 방식 확정)로 진행 예정
- **설계 기준**: 매일 5분만 투자해도 지속 가능할 것

---

## 0. 왜 이 순서로 설계하는가

바로 개발부터 시작하면 십중팔구 "그럴싸한 AI 콘텐츠 자판기"가 나온다.
그건 이미 시중에 널렸고, 목적(철학 자산 축적)과 정반대다.

먼저 **"무엇을 정의해야 이 시스템이 실제로 김효정을 담게 되는가"**를 MECE로 쪼개고,
어떤 순서로 쌓아야 중간에 무너지지 않는지 로드맵을 짠 뒤 — 그 다음에야 개발한다.

## 1. MECE 구성 요소 (9개 · 3개 층)

### A. 기반층 (Foundation) — 시스템이 "누구를 흉내낼지"의 기준
| 항목 | 정의 | 왜 필요한가 |
|---|---|---|
| **Identity** | 가치관·신념·톤·취향·의사결정 기준의 명시적 프로필 | 이게 없으면 뒤 단계 전부가 "일반적으로 그럴듯한 AI 말투"로 수렴함. 모든 단계가 참조하는 단일 소스 |

### B. 핵심 파이프라인 (Core Pipeline) — 생각이 자산이 되기까지의 흐름
| 항목 | 정의 | 왜 필요한가 |
|---|---|---|
| **Thinking (캡처)** | 일상적 생각·메모·아이디어를 마찰 없이 즉시 기록하는 입구 | 재료가 없으면 뒤에 아무것도 못 만듦. 가장 자주 실패하는 지점 = 여기서 귀찮아서 안 씀 |
| **Memory (기억)** | 캡처된 생각을 나중에 찾고 엮을 수 있게 최소한으로 구조화·저장 | 기록만 쌓이고 검색·재활용이 안 되면 "일기장"에서 멈춤. 철학으로 못 올라감 |
| **Philosophy (철학화)** | 반복되는 생각 패턴을 발견해 원칙/관점으로 승화시키는 리뷰 프로세스 | 이 시스템의 핵심 차별점. 없으면 그냥 메모 앱 |
| **Writing (문체 변환)** | 철학·생각을 김효정 문체로 글(에세이·뉴스레터·카피)로 변환 | 철학이 텍스트 자산이 되는 지점. `voice-dna` 스킬과 직결 |
| **Visual (비주얼 변환)** | 철학·문체를 디자인 취향에 맞는 이미지·무드보드로 변환 | 실장님 본업(평면 디자인)과 직결되는 표현 채널 |
| **Publishing (배포)** | 완성된 글/이미지를 실제 채널(뉴스레터·SNS·개인 사이트 등)에 내보내는 규칙 | 산출물이 로컬에만 쌓이면 브랜드 자산이 아니라 파일 더미 |

### C. 메타층 (Meta) — 위 파이프라인을 오래 돌아가게 하는 것
| 항목 | 정의 | 왜 필요한가 |
|---|---|---|
| **Brand (자산화)** | 축적된 글·이미지·철학을 포트폴리오/아카이브로 정리해 재사용 가능하게 관리 | 시간이 지나도 "그때 그거 어디 있더라"가 안 되게. 복리로 쌓이는 자산의 창고 |
| **Workflow (운영 리듬)** | 언제·어떻게 위 단계들이 트리거되는지의 자동화·스케줄 설계 | 사람이 매번 기억해서 실행하면 100% 며칠 안에 멈춤. 단, **맨 마지막에** 설계 (아래 이유 참고) |

## 2. 빌드 로드맵 (권장 순서 · 9단계)

> 원칙: **재료(Identity) → 입구(Thinking) → 저장(Memory) → 정제(Philosophy) → 표현(Writing/Visual) → 배포(Publishing) → 창고(Brand) → 자동화(Workflow)**
> 자동화를 맨 뒤에 두는 이유: 사람이 수동으로 몇 바퀴 돌려서 프로세스가 검증되기 전에 자동화하면, "잘못된 프로세스를 빠르게 반복"하게 될 뿐임.

1. **Identity 정의** — 가치관·문체·취향을 최소 세트로 명문화 (`voice-dna` 스킬 활용 가능). *가장 먼저인 이유*: 이후 모든 판단(뭘 기록할지, 뭘 철학으로 볼지, 어떤 톤으로 쓸지)의 기준선이 됨.
2. **Thinking 캡처 방식 확정** — 하루 5분 안에 끝나는 입력 창구 하나만 정한다 (예: 음성메모, 슬랙 DM, 노션 퀵노트 중 택1). *왜 하나만*: 창구가 여러 개면 매일 안 씀 → 시스템 전체가 멈춤.
3. **Memory 최소 구조 설계** — 캡처된 원문에 최소 태그(주제/감정/날짜) 하나만 붙여 검색 가능하게. *처음엔 화려한 DB·임베딩 불필요*: 과설계는 5분 원칙을 깬다.
4. **Philosophy 리뷰 루틴 설계** — 주간/월간 단위로 쌓인 생각을 훑어 반복 패턴 → 원칙 초안으로 승화. *일간이 아니라 주간인 이유*: 매일 통찰을 뽑으라는 건 부담이 커서 지속 불가능. 캡처는 매일, 정제는 배치로.
5. **Writing 템플릿 구축** — 철학 초안이 몇 개 쌓이면 그걸 글로 바꾸는 템플릿·톤 규칙 확정 (`voice-dna`+`anti-ai-writing` 재사용). *4번 이후인 이유*: 원재료(철학) 없이 템플릿부터 만들면 또 "그럴듯한 AI 글"이 나옴.
6. **Visual 스타일 가이드 구축** — 문체가 안정되면 같은 관점을 이미지·무드보드로 확장. *5번 이후인 이유*: 텍스트로 먼저 관점이 명확해져야 비주얼이 산으로 안 감 (기존 `concept-director` 에이전트의 "관점 먼저" 원칙과 동일).
7. **Publishing 채널·주기 확정** — 어디에 얼마나 자주 내보낼지 결정 (완벽한 자동화 없이 수동 발행부터). *왜 아직 수동*: 채널·주기 자체가 검증 전이라 자동화 대상이 아직 아님.
8. **Brand 아카이브 구조 확립** — 나온 결과물을 검색·재사용 가능한 자산 창고로 정리 (`.claude/library/` 확장 또는 별도 아카이브). *7번 이후인 이유*: 발행 이력이 쌓여야 아카이브 구조의 실제 필요가 드러남.
9. **Workflow 자동화** — 1~8단계가 최소 1~2달 수동으로 안정적으로 돌아간 뒤, 트리거·스케줄로 자동화. *맨 마지막인 이유*: 위에서 설명한 대로, 검증 안 된 프로세스의 자동화는 오히려 리스크.

**일간 부담 설계**: 매일 하는 건 **2단계(캡처)뿐** — 5분 이내. 3~9단계는 전부 주간/월간 배치 또는 일회성 설정. 이게 "매일 5분" 원칙을 실제로 지키는 방법.

## 3. 다음 단계로 넘어가기 위해 필요한 질문 (인터뷰)

아래 답이 있어야 **1단계(Identity)**부터 실제로 정의를 시작할 수 있음. 추측하지 않고 확인 필요.

1. **캡처 창구**: 하루 5분 기록을 어디서 하고 싶으신가요? (예: 음성메모 → 텍스트 변환 / 슬랙 DM / 노션 퀵노트 / 기타)
2. **Identity 재료**: 이미 써둔 글·메모·인터뷰·SNS 글 중 "이게 내 생각·문체다" 싶은 게 있나요? (voice-dna처럼 20개 정도 있으면 바로 프로파일링 가능)
3. **철학화 범위**: 이 시스템이 다루길 원하는 생각의 범위가 디자인·업무 관련인가요, 개인적 삶의 철학까지 포함하나요?
4. **최종 산출 목적**: 축적된 걸 어디에 쓰고 싶으신가요? (개인 브랜드 콘텐츠 발행용 / 사내 크리에이티브 디렉션 기준 문서용 / 둘 다 / 아직 미정)
