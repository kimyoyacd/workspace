# Project Discovery — 프로젝트 디스커버리 · 브랜드 팩트북 · 문제 재정의

marketer 파이프라인의 STEP 1~3. 시장조사보다 먼저 **프로젝트와 브랜드를 이해하고 진짜 문제를 정의**한다.

## 산출 파일

```
.claude/projects/YYYYMM_PROJECT/outputs/
├── 01_project_discovery.md
├── 02_brand_factbook.md
└── 03_problem_definition.md
```

## 정보 상태 표시 (전 산출물 공통)

모든 정보에 상태를 구분해 표기한다:

| 상태 | 의미 |
|---|---|
| `FACT` | 공식 자료나 내부 문서로 확인된 사실 |
| `CLIENT CLAIM` | 클라이언트가 전달했지만 외부 검증되지 않은 내용 |
| `INFERENCE` | 확인된 사실을 바탕으로 도출한 해석 |
| `HYPOTHESIS` | 추가 검증이 필요한 가설 |
| `UNKNOWN` | 현재 확인할 수 없는 내용 |

정보가 부족하면 임의로 채우지 않는다. 대신 기록한다:
현재 확인된 내용 / 확인되지 않은 내용 / 확인이 필요한 이유 / 클라이언트에게 물어볼 질문 / 외부 조사가 필요한 항목.

## STEP 2 — 브랜드 팩트북 (02_brand_factbook.md)

단순 소개문이 아니라, **이후 문제 정의와 시장조사의 기준이 되는 브랜드 사전 이해 문서**.

### 브랜드 목적과 역할

고객이 선택해야 할 이유 · 브랜드가 만들고 싶은 미래

### 제품·서비스·경험

경쟁 대안 · 강점 · 약점

### 타깃 (인구통계가 아니라 행동·욕구·선택 기준·장벽 중심)

## STEP 3 — 목표와 문제 재정의 (03_problem_definition.md)

팩트북 이후 네 가지를 비교한다:

| 관점 | 내용 |
|---|---|
| `Brand Says` | 브랜드가 공식적으로 주장하는 것 |
| `Brand Does` | 제품·서비스·공간에서 실제로 제공하는 것 |
| `Customer Sees` | 고객이 실제로 경험하고 인식하는 것 |
| `Market Compares` | 경쟁사와 비교했을 때 보이는 위치 |

이를 기반으로 도출:
현재 상태 · 희망 상태 · 활용 가능한 브랜드 자산 · 해결해야 할 브랜드 격차 · 검증되지 않은 가설 ·
표면적인 클라이언트 요청 · 요청 이면의 사업 목적 · **진짜 해결해야 할 문제** ·
최종적으로 내려야 할 의사결정 · 프로젝트 성공 기준 · 이후 리서치 질문

### 문제 정의 형식 — 현상이 아니라 의사결정 질문으로

```
표면적 요청: 젊은 고객이 좋아할 새로운 브랜드 디자인이 필요하다.
현재 현상: 기존 브랜드가 젊은 고객에게 오래된 브랜드로 인식된다.
원인 가설: 브랜드가 가진 전문성과 제품 가치를 새로운 고객이 체감할 수 있는 경험과 언어로 전환하지 못하고 있다.
진짜 해결 문제: 기존의 신뢰 자산을 훼손하지 않으면서, 새로운 고객이 이 브랜드를 자신의 선택으로
받아들이도록 가치를 재해석하고 증명하는 방식은 무엇이어야 하는가?
```

## 완료 기준

- [ ] 프로젝트 성격이 정의되었는가
- [ ] 브랜드 팩트북이 작성되었는가
- [ ] 타깃이 행동·욕구 기준으로 정의되었는가
- [ ] FACT/CLIENT CLAIM/INFERENCE/HYPOTHESIS/UNKNOWN이 구분됐는가
- [ ] 현재 상태와 목표 상태가 비교됐는가
- [ ] 진짜 해결 문제가 의사결정 질문으로 정의됐는가
