# 브랜드 가이드라인 덱 제작

디자인 외주 실장이 클라이언트 프로젝트마다 반복 제작하는 **한글 브랜드 가이드라인 문서**를 표준 구조로 빠르게 만들기 위한 스킬. TEMS Intelligence 프로젝트(`.claude/projects/202607_TEMS브랜딩_가이드_v2.html`)에서 완성했고, 이후 모든 브랜딩 프로젝트에 재사용한다.

> v1(18페이지, 구조 위주)을 클라가 "피그마 템플릿에 있는 밸류·브랜드 파운데이션·이미저리·머치 응용을 다 누락했다"고 반려한 뒤 v2(23페이지)로 보강했다. **섹션 개수만 채우고 실제 템플릿 카피 패턴·이미저리·응용 깊이를 스킵하면 부족하다고 판단된다** — 아래 "언제 쓰나" 다음 항목을 반드시 지킬 것.

## 언제 쓰나
- 로고·컬러·타이포 등 BI 요소가 **확정**된 뒤, 클라이언트에게 전달할 정식 가이드 문서가 필요할 때
- 참고할 피그마 브랜드 가이드 템플릿 링크가 있을 때(구조 참고용 — 내용을 그대로 베끼지 않는다)
- 심볼이 아직 미확정이면 이 스킬을 실행하지 말고, 먼저 심볼부터 확정할 것 (명분 없는 가이드 문서는 무의미)

## 선행 조건 (반드시 확인)
1. **브랜드 전략이 확정돼 있는가?** 최소한 아래가 있어야 한다 — 없으면 먼저 만들라고 사용자에게 요청:
   - 브랜드명 / 하는 일 / 타깃 / 산업군 / 브랜드 인상
   - 미션 · 비전 · 포지셔닝 스테이트먼트
   - 브랜드 핵심가치(Pillars) 5개 — **한 단어 + 짧고 강한 한 문장**(예: "맥락 — 정답이 아니라 이유를 본다"). 키워드 단독이나 반대로 긴 설명 문단 둘 다 아니다 — 실제 브랜드 가이드 템플릿 카피는 항상 이 리듬이다.
   - 브랜드 퍼스낼리티(형용사 8~10개)
   - 톤앤보이스 Do/Don't
2. **심볼/로고가 확정돼 있는가?** 벡터(SVG) 형태로 있어야 한다. 래스터 이미지만 있으면 트레이싱부터 하고, 트레이싱 결과물엔 반드시 "래스터 기반 추정 — 원본 벡터 수신 시 교체" 주석을 남긴다. 확정 안 된 상태로 가이드를 만들지 말 것 — 심볼이 나중에 바뀌면 문서 전체를 다시 만들어야 한다.
3. **참고 피그마 템플릿이 있으면** `mcp__Figma__get_metadata`로 전체 페이지 목록을 훑고, **그 중 대표 페이지 3~5개는 반드시 `get_design_context`로 실제 카피까지 읽는다** — 구조만 보고 넘어가지 말 것. 특히 "Values/Foundation" 계열 페이지는 실제 문안을 확인해야 진짜 톤을 알 수 있다(예: TEMS 템플릿의 Brand Values는 "제목+긴 설명"이 아니라 **"한 단어 + 짧고 강한 한 문장"** 패턴이었다 — `Leadership — The courage to shape a better future.` 이 패턴을 놓치면 카피가 늘어지고 힘이 빠진다). 이미지·문구를 그대로 복사하지 않고 톤·구조·리듬만 가져와 브랜드에 맞게 새로 쓴다.
4. **템플릿에 Imagery/무드보드, Merch(굿즈) 응용 페이지가 있으면 절대 생략하지 않는다.** "심볼이 미확정이라 나중에" 식으로 통째로 빼면 클라이언트가 "다 누락했다"고 느낀다. 심볼이 미확정이어도 최소한 자리는 만들고, 심볼이 확정됐다면 실제로 채운다.

## 표준 섹션 구조 (23페이지 기준, 프로젝트 규모에 따라 가감)
1. **Cover** — 심볼 + "Brand Identity Guidelines" + 버전·날짜
2. **Welcome** — 브랜드를 한 문장으로 압축한 큰 헤드라인 + 2줄 소개(장황한 설명문 금지, 임팩트 우선)
3. **Table of Contents** — 5대 섹션(Strategy/Logo/Color&Type/Imagery/Applications) 요약
4. **Section Cover: Brand Strategy**
5. **Defining [Brand]** — "브랜드란 무엇인가"를 정의하는 짧은 인트로 2문단 + 1인칭 브랜드 내러티브("우리는 ~다") 큰 문단. 여기가 클라가 가장 자주 찾는 페이지 — 빠뜨리면 "정의가 없다"고 느낀다.
6. Mission · Vision · Positioning
7. Brand Values — **한 단어(굵게, 크게) + 짧고 강한 한 문장**의 리스트. 문단 설명 금지(위 "선행 조건" 참고).
8. Brand Personality (형용사 8~10개) + Tone of Voice (Do/Don't)
9. **Section Cover: Logo**
10. Logo Construction — 심볼이 왜 이 형태인지 논리 (브랜드 핵심가치와 반드시 연결지어 설명)
11. Clear Space & Lockup — 가로/세로 락업 + 클리어스페이스 그리드(점선 박스, x단위 표기) + 락업 간격
12. Minimum Size — 인쇄/디지털 최소 크기, 파비콘 전용 락업 안내
13. Logo Color Variations — 라이트/다크 배경, 그라데이션 배경 예외 규칙
14. **Section Cover: Color & Typography**
15. Color System — 로고용 모노크롬 + (있다면) 브랜드 월드용 확장 팔레트, 역할을 명확히 분리해서 설명
16. Typography — 워드마크 + 본문 서체, 커스텀 서체 여부 명시
17. **Section Cover: Imagery**
18. Moodboard/Photo — 브랜드 세계관을 보여주는 무드 사진 1장(풀블리드 + 스크림 그라데이션 + 캡션 카피). 사진이 없으면 Higgsfield `generate_image`로 생성(아래 "이미저리·머치 제작법" 참고) — 생략하지 않는다.
19. **Section Cover: Applications**
20. 명함 (앞/뒤, 라이트/그라데이션 버전 등)
21. Merch — 티셔츠·텀블러 등 굿즈 목업 2종. "가상의 회사 분위기"를 체감시키는 페이지 — 클라가 명시적으로 요청하지 않아도 기본 포함 권장.
22. App Icon — 폰 홈 화면 목업
23. **Next Steps** — 미확정 항목, 다음 버전에서 보완할 것

섹션 4/9/14/17/19처럼 색이 있는 "Section Cover" 페이지는 그라데이션이나 브랜드 다크 컬러의 풀블리드 배경 + 큰 타이틀 하나만 놓는다(내용 없음, 챕터 구분용).

## 이미저리·머치 제작법 (사진이 없을 때)
클라가 무드 레퍼런스 사진을 인라인으로 첨부해도 파일로 저장되지 않는 경우가 흔하다(트레이싱 불가). 이런 사진은 "이 느낌으로 만들어달라"는 방향 지시로 받아들이고, 실제 이미지는 Higgsfield `generate_image`로 새로 생성한다:
- **오피스/무드 사진**: `nano_banana_2`(또는 `nano_banana_pro`) 모델, 브랜드 컬러 톤을 프롬프트에 명시(예: "warm coral to deep indigo gradient light"), 사람·텍스트·로고 없는 순수 공간/오브젝트 샷으로 요청
- **머치 목업(티셔츠·텀블러 등)**: `marketing_studio_image` 모델로 **로고 없는 빈 제품 스튜디오컷**을 생성한 뒤, 실제 로고 SVG를 CSS `position:absolute`로 정확한 위치에 오버레이한다 — AI가 로고 자체를 그리게 하면 형태가 틀어진다. 절대 생성 모델에게 우리 로고를 직접 그리게 하지 않는다.
- 생성된 이미지는 Bash로 다운로드 후 Pillow 등으로 리사이즈(가로 600~900px)·JPEG 압축(quality ~78)해서 base64 데이터 URI로 인라인 삽입한다(Artifact는 외부 URL을 막으므로 자체 완결형이어야 함). 원본 PNG가 수백 KB~수 MB면 반드시 리사이즈 후 인코딩할 것 — 안 그러면 파일이 비대해진다.
- 대량 base64(수십~수백 KB 텍스트)를 Write 도구 프롬프트에 직접 타이핑하지 말고, `{{PLACEHOLDER}}` 토큰을 넣은 템플릿 HTML을 먼저 쓴 뒤 Python/Bash로 문자열 치환해 최종 파일을 조립한다.

## 페이지 템플릿 (재사용 CSS 패턴)
모든 페이지는 16:9 비율 카드, 흰 배경(또는 다크/그라데이션 섹션 커버), 공통 헤더/푸터:
```
헤더: [브랜드명 볼드] [섹션명 회색]  ........................  [페이지번호]
      ─────────────────────────────────────── (얇은 구분선)
본문: (섹션별 콘텐츠)
푸터: [브랜드명 · 섹션명]  ........................  [페이지번호]
      (얇은 구분선 위)
```
`aspect-ratio:16/9`인 `.page` div를 세로로 나열해 스크롤 덱으로 보여준다(실제 인쇄/PDF 변환은 다음 단계). 다크 섹션 커버는 `.page.dark`(잉크 단색) 또는 `.page.gradbg`(브랜드 그라데이션 방사형) 클래스로 구분.

작업 시 TEMS 프로젝트의 `_가이드_v2.html`을 뼈대로 복제한 뒤 브랜드명·색상·심볼 path·전략 텍스트·이미지만 치환하는 방식을 권장 — CSS 변수(`--ink`, `--gray*`, 브랜드 액센트 컬러)만 바꾸면 새 브랜드에 맞는 톤으로 전환된다.

## 산출 후 처리
1. Playwright로 렌더링해 실제 스크린샷으로 레이아웃 깨짐(텍스트 겹침, 심볼 오배치 등)을 반드시 확인한다 — 코드만 보고 완성됐다고 판단하지 말 것.
2. Artifact로 발행해 사용자가 바로 볼 수 있게 한다.
3. 프로젝트의 버전 정책을 따른다 — 기존 파일을 덮어쓰지 않고 `_가이드_v1.html` → `_가이드_v2.html` 순으로 새 파일 저장(CLAUDE.md 및 프로젝트 노트에 버전 정책이 명시돼 있으면 그것을 따른다).
4. 프로젝트 마스터 노트(`.claude/projects/<프로젝트>.md`)에 가이드 문서 제작 사실과 버전을 기록한다.
