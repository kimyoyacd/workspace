# 브랜드 가이드라인 덱 제작

디자인 외주 실장이 클라이언트 프로젝트마다 반복 제작하는 **한글 브랜드 가이드라인 문서**를 표준 구조로 빠르게 만들기 위한 스킬. TEMS Intelligence 프로젝트(`.claude/projects/202607_TEMS브랜딩_가이드_v3.html`)에서 완성했고, 이후 모든 브랜딩 프로젝트에 재사용한다.

> 3라운드에 걸쳐 반려당하며 다듬어졌다 — 매번 "실제 템플릿을 제대로 안 보고 대충 흉내만 냈다"는 지적이었다:
> - v1(18p, 구조만): "밸류·브랜드 파운데이션·이미저리·머치를 다 누락했다"
> - v2(23p, 내용 보강): "Page Header 크기가 실제 피그마와 다르다, Social Media Content가 빠졌다, 무드 사진 반영 안 했다"
> - v3(28p): 실제 피그마 컴포넌트를 `get_design_context`로 읽어 헤더 규격을 그대로 이식, Social Media Profile·Logo Don'ts·Primary Logo 신설, 클라 무드 사진의 색을 추정 반영
> **교훈: "표준 구조를 따라간다"는 것은 섹션 이름만 비슷하게 짓는 게 아니라, 실제 참고 자료(피그마 컴포넌트 치수·폰트·카피 패턴)를 열어서 수치까지 베끼는 수준으로 맞추는 것이다.** 아래 "언제 쓰나" 다음 항목을 반드시 지킬 것.

## 언제 쓰나
- 로고·컬러·타이포 등 BI 요소가 **확정**된 뒤, 클라이언트에게 전달할 정식 가이드 문서가 필요할 때
- 참고할 피그마 브랜드 가이드 템플릿 링크가 있을 때(구조 참고용 — 내용을 그대로 베끼지 않는다)
- 심볼이 아직 미확정이면 이 스킬을 실행하지 말고, 먼저 심볼부터 확정할 것 (명분 없는 가이드 문서는 무의미)

## 선행 조건 (반드시 확인)
1. **브랜드 전략이 확정돼 있는가?** 최소한 아래가 있어야 한다 — 없으면 먼저 만들라고 사용자에게 요청:
   - 브랜드명 / 하는 일 / 타깃 / 산업군 / 브랜드 인상
   - 미션 · 비전 · 포지셔닝 스테이트먼트
   - 브랜드 핵심가치(Pillars) 5개 — **한 단어 + 짧고 강한 한 문장**(예: "맥락 — 정답이 아니라 이유를 본다"). 키워드 단독이나 반대로 긴 설명 문단 둘 다 아니다 — 실제 브랜드 가이드 템플릿 카피는 항상 이 리듬이다.
   - 브랜드 퍼스낼리티(형용사 8~10개)
   - 톤앤보이스 Do/Don't
2. **심볼/로고가 확정돼 있는가?** 벡터(SVG) 형태로 있어야 한다. 래스터 이미지만 있으면 트레이싱부터 하고, 트레이싱 결과물엔 반드시 "래스터 기반 추정 — 원본 벡터 수신 시 교체" 주석을 남긴다. 확정 안 된 상태로 가이드를 만들지 말 것 — 심볼이 나중에 바뀌면 문서 전체를 다시 만들어야 한다.
3. **참고 피그마 템플릿이 있으면 `mcp__Figma__get_metadata`로 페이지 목록 전체를 훑고, 대표 페이지 3~5개는 반드시 `get_design_context`로 실제 코드·카피·수치까지 읽는다.** "구조만 보고 넘어가는 것"이 세 번의 반려 중 두 번의 근본 원인이었다. 확인할 것:
   - **카피 리듬**: "Values/Foundation" 계열은 문안을 봐야 톤을 안다(예: `Leadership — The courage to shape a better future.` — 한 단어+짧고 강한 문장. 긴 설명 문단이 아니다).
   - **Page Header 컴포넌트 정확한 스펙**: 대부분의 브랜드 가이드 템플릿은 헤더가 라벨 한 줄짜리가 아니라 **2행 구조**다 — ①Page Number 행(문서 타이틀 + 섹션명·페이지번호, 작은 글씨) ②Headline+Paragraph 행(그 페이지의 실제 제목 큰 글씨 + 설명 1~2단). "헤더"를 작은 라벨로만 처리하면 실제 템플릿과 크기가 달라진다는 지적을 받는다 — `get_design_context`로 실측한 폰트 크기·행높이·컬럼 폭을 그대로 비율 환산해서 쓸 것(아래 "Page Header 표준 스펙" 참고).
   - 이미지·문구를 그대로 복사하지 않고 톤·구조·리듬·치수만 가져와 브랜드에 맞게 새로 쓴다.
4. **템플릿에 있는 페이지 유형은 하나도 스킵하지 않는다.** Imagery/무드보드, Merch(굿즈), **Social Media Profile/Content**(앱 아이콘 + 인스타그램식 프로필 목업), Logo Don'ts, Primary Logo 단독 쇼케이스, Typography가 여러 페이지로 나뉘어 있으면 그대로 나눈다 — "심볼이 미확정이라 나중에" 식으로 통째로 빼면 클라이언트가 "다 누락했다"고 느낀다. 심볼이 미확정이어도 최소한 자리는 만들고, 확정됐다면 실제로 채운다.
5. **클라가 새 무드 레퍼런스(색감·톤)를 추가로 던지면 "혼합"으로 받아들인다.** 기존 확정 브랜드 컬러를 통째로 갈아엎지 말고, 용도를 분리한 두 번째 그라데이션/팔레트로 추가해 Cover·Welcome·Section Cover 등 표지류에 적용하고, 로고 자체의 모노크롬 원칙은 유지한다(예: TEMS는 "Discovery Gradient"(로고·마케팅용, 기존)와 "Atmosphere Gradient"(표지·배경용, 신규)를 병기).

## 표준 섹션 구조 (28페이지 기준, 프로젝트 규모에 따라 가감)
1. **Cover** — 심볼 + "Brand Identity Guidelines" + 버전·날짜. 배경은 브랜드 표지톤(무드 블렌딩 대상 1순위).
2. **Welcome** — 브랜드를 한 문장으로 압축한 큰 헤드라인 + 2줄 소개(장황한 설명문 금지, 임팩트 우선). 표지와 같은 톤.
3. **Table of Contents** — 전체 섹션 요약 (개수는 실제 구성에 맞춰)
4. **Section Cover: Brand Strategy**
5. **Defining [Brand]** — "브랜드란 무엇인가"를 정의하는 짧은 인트로 2문단 + 1인칭 브랜드 내러티브("우리는 ~다") 큰 문단. 클라가 가장 자주 찾는 페이지 — 빠뜨리면 "정의가 없다"고 느낀다.
6. Mission · Vision · Positioning
7. Brand Values — **한 단어(굵게, 크게) + 짧고 강한 한 문장**의 리스트. 문단 설명 금지.
8. Brand Personality (형용사 8~10개) + Tone of Voice (Do/Don't)
9. **Section Cover: Logo**
10. **Primary Logo** — 심볼+워드마크 기본 락업을 라이트/다크 배경 나란히 크게 보여주는 단독 쇼케이스. 구성 논리 설명 전에 "이게 우리 로고다"를 먼저 각인시키는 페이지 — 생략하면 허전하다.
11. Logo Construction — 심볼이 왜 이 형태인지 논리 (브랜드 핵심가치와 반드시 연결지어 설명)
12. Clear Space & Lockup — 가로/세로 락업 + 클리어스페이스 그리드(점선 박스, x단위 표기) + 락업 간격
13. Minimum Size — 인쇄/디지털 최소 크기, 파비콘 전용 락업 안내
14. Logo Color Variations — 라이트/다크/그라데이션 배경 예외 규칙
15. **Logo Don'ts** — 비율 왜곡·임의 배색·회전·이펙트 추가·저대비 배경·레터폼 재제작 등 5~6개 위반 사례를 실제로 렌더링해서 보여준다. 대부분의 실제 가이드에 있는데 놓치기 쉬운 페이지.
16. **Section Cover: Color & Typography**
17. Color System — 로고용 모노크롬 + 브랜드 월드용 그라데이션(들). 용도별로 이름 붙여 역할을 분리(예: "Discovery Gradient"=로고·마케팅, "Atmosphere Gradient"=표지·배경).
18. Typography — Primary(워드마크)
19. Typography — Secondary(국문 본문 서체 + 라벨/수치용 모노스페이스). 템플릿이 타이포를 여러 페이지로 나눴으면 따라서 나눈다 — 한 페이지에 욱여넣지 않는다.
20. **Section Cover: Imagery**
21. Moodboard/Photo — 브랜드 세계관을 보여주는 무드 사진(풀블리드 + 스크림 그라데이션 + 캡션 카피). 사진이 없으면 Higgsfield `generate_image`로 생성.
22. **Mood Mix** — 클라가 새 무드 레퍼런스를 줬다면, 그 레퍼런스의 레이아웃 구조(예: 플로팅 콘텐츠 카드+헤드라인+CTA+컬러칩)를 그대로 재현하되 내용은 브랜드 것으로 채운 페이지. "혼합해서 써달라"는 요청에 가장 직접적으로 답하는 페이지.
23. **Section Cover: Applications**
24. 명함 (앞/뒤, 라이트/그라데이션 버전 등)
25. Merch — 티셔츠·텀블러 등 굿즈 목업. "가상의 회사 분위기"를 체감시키는 페이지 — 클라가 명시적으로 요청하지 않아도 기본 포함 권장.
26. App Icon — 폰 홈 화면 목업
27. **Social Media Profile** — 앱 아이콘 + 인스타그램/LinkedIn 스타일 프로필 목업(아바타·팔로워 수·바이오·피드 그리드) + 운영 방향 카피(주 채널, 캡션 톤, 이모지/해시태그 규칙). "SNS 어떻게 운영할지 보여달라"는 요청에 대응하는 표준 페이지 — 미리 넣어둘 것.
28. **Next Steps** — 미확정 항목, 다음 버전에서 보완할 것

섹션 4/9/16/20/23처럼 색이 있는 "Section Cover" 페이지는 그라데이션이나 브랜드 컬러의 풀블리드 배경 + 큰 타이틀 하나만 놓는다(내용 없음, 챕터 구분용).

## Page Header 표준 스펙 (반드시 실측치로 맞출 것)
브랜드 가이드 템플릿의 헤더는 대개 1920×1080 캔버스 기준 아래 구조다 — 자기 덱의 페이지 폭 대비 **동일 비율**로 환산해서 쓴다(임의로 작게 만들지 말 것):
```
[Page Number 행 — 전체 높이 대비 얇게, mono 폰트]
  Title(문서 전체 이름, 폭 ~31%, border-bottom, padding 8px) | Section명 ..... 페이지번호 (flex:1, border-bottom)
[Headline+Paragraph 행 — 본 스펙의 핵심]
  Headline(그 페이지 제목, 폭 ~31%, bold, 60/56px 비율 = 페이지 폭의 약 3%) | Paragraph 1(flex:1, mono/sans 20/28px 비율) | Paragraph 2(flex:1, 동일)
```
전체 헤더 높이는 페이지 세로 길이의 약 19~20%(210/1080)를 차지한다 — 라벨 한 줄짜리 얇은 헤더로 축소하지 말 것. 국문 프로젝트에서는 Paragraph 영역만 가독성을 위해 sans-serif(Pretendard 등)로 바꾸고, Page Number 행과 수치·영문 라벨은 모노스페이스를 유지해 원본의 "정밀한 인상"을 살린다.

## Figma 소스가 "빈 템플릿"인지 확인
클라가 준 링크가 자기들이 이미 채워 넣은 실제 파일이 아니라 배포용 빈 템플릿(예: fileKey는 같지만 마스터 컴포넌트만 있고 브랜드명이 플레이스홀더인 경우)일 수 있다. `get_metadata`로 최상위 페이지들("Pages"/"Logo & Icons"/"Components" 등)을 확인했을 때 심볼·문구가 여전히 제네릭하면 — 클라가 스크린샷으로 보여준 "이미 완성된 페이지"는 다른(비공개) 파일에 있다는 뜻이다. 이 경우 솔직하게 "지금 주신 링크는 빈 템플릿이라 구조·수치만 가져왔고, 실제 완성 파일이 따로 있다면 그 URL을 주셔야 한다"고 밝힌다 — 추측으로 "그대로 가져왔다"고 우기지 않는다.

## 이미저리·머치 제작법 (사진이 없을 때)
클라가 무드 레퍼런스 사진을 인라인으로 첨부해도 파일로 저장되지 않는 경우가 흔하다(트레이싱 불가). 이런 사진은 "이 느낌으로 만들어달라"는 방향 지시로 받아들이고, 실제 이미지는 Higgsfield `generate_image`로 새로 생성한다:
- **오피스/무드 사진**: `nano_banana_2`(또는 `nano_banana_pro`) 모델, 브랜드 컬러 톤을 프롬프트에 명시(예: "warm coral to deep indigo gradient light"), 사람·텍스트·로고 없는 순수 공간/오브젝트 샷으로 요청
- **머치 목업(티셔츠·텀블러 등)**: `marketing_studio_image` 모델로 **로고 없는 빈 제품 스튜디오컷**을 생성한 뒤, 실제 로고 SVG를 CSS `position:absolute`로 정확한 위치에 오버레이한다 — AI가 로고 자체를 그리게 하면 형태가 틀어진다. 절대 생성 모델에게 우리 로고를 직접 그리게 하지 않는다.
- 생성된 이미지는 Bash로 다운로드 후 Pillow 등으로 리사이즈(가로 600~900px)·JPEG 압축(quality ~78)해서 base64 데이터 URI로 인라인 삽입한다(Artifact는 외부 URL을 막으므로 자체 완결형이어야 함). 원본 PNG가 수백 KB~수 MB면 반드시 리사이즈 후 인코딩할 것 — 안 그러면 파일이 비대해진다.
- 대량 base64(수십~수백 KB 텍스트)를 Write 도구 프롬프트에 직접 타이핑하지 말고, `{{PLACEHOLDER}}` 토큰을 넣은 템플릿 HTML을 먼저 쓴 뒤 Python/Bash로 문자열 치환해 최종 파일을 조립한다.

## 페이지 템플릿 (재사용 CSS 패턴)
모든 페이지는 16:9 비율 카드, 흰 배경(또는 다크/그라데이션 섹션 커버), 공통 헤더/푸터:
```
헤더: [브랜드명 볼드] [섹션명 회색]  ........................  [페이지번호]
      ─────────────────────────────────────── (얇은 구분선)
본문: (섹션별 콘텐츠)
푸터: [브랜드명 · 섹션명]  ........................  [페이지번호]
      (얇은 구분선 위)
```
`aspect-ratio:16/9`인 `.page` div를 세로로 나열해 스크롤 덱으로 보여준다(실제 인쇄/PDF 변환은 다음 단계). 다크 섹션 커버는 `.page.dark`(잉크 단색) 또는 `.page.gradbg`(브랜드 그라데이션 방사형) 클래스로 구분.

작업 시 TEMS 프로젝트의 `_가이드_v3.html`을 뼈대로 복제한 뒤 브랜드명·색상·심볼 path·전략 텍스트·이미지만 치환하는 방식을 권장 — CSS 변수(`--ink`, `--gray*`, 브랜드 액센트 컬러)만 바꾸면 새 브랜드에 맞는 톤으로 전환된다. `.ph`/`.ph-row1`/`.ph-row2`/`.ph-title`/`.ph-section`/`.ph-headline`/`.ph-para` 클래스가 위 "Page Header 표준 스펙"을 구현한 재사용 가능한 헤더 파셜이다.

## 산출 후 처리
1. Playwright로 렌더링해 실제 스크린샷으로 레이아웃 깨짐(텍스트 겹침, 심볼 오배치 등)을 반드시 확인한다 — 코드만 보고 완성됐다고 판단하지 말 것.
2. Artifact로 발행해 사용자가 바로 볼 수 있게 한다.
3. 프로젝트의 버전 정책을 따른다 — 기존 파일을 덮어쓰지 않고 v1→v2→v3… 순으로 새 파일 저장(CLAUDE.md 및 프로젝트 노트에 버전 정책이 명시돼 있으면 그것을 따른다).
4. 프로젝트 마스터 노트(`.claude/projects/<프로젝트>.md`)에 가이드 문서 제작 사실과 버전, 그리고 **이번에 클라가 지적한 구체적 피드백**(다음 프로젝트에도 재발할 수 있는 패턴이면)을 기록한다.
