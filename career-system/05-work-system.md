# 05. 업무 운영 시스템 — 파일·폴더 정리

> 출발점: "여러 일을 동시에 하니 폴더가 안 잡힌다" = 게으름이 아니라 **구조의 부재**.
> 동시 진행이 많을수록 "모든 게 들어갈 정해진 집"이 필요하다.

마지막 갱신: 2026-06 (설계)

---

## 진단

- 보관: 로컬 + Google Drive + 클라우드 **혼합**
- 프로젝트 관리: **노션**(상태 체크) / 리소스: **시트**
- 주력 작업 3종: **제안서 · 회사소개 · 견적** (= "일을 따내는 도구" = 강점인 '맥락 포장'이 발휘되는 곳)
- 문제: 동시 진행 多 → 일관된 보관 위치 없음 → 흩어짐

## 빼기 원칙 5가지

1. **상태는 딱 3개** — `진행중` / `보관` / `자산`. "나중에 볼 것" 폴더 금지(=쓰레기장).
2. **한 프로젝트 = 한 집**, 내부 구조는 항상 동일하게 반복(뇌가 길을 외움).
3. **"최신본은 단 하나"** — 회사소개·제안서 버전 지옥 종료. `_최신본` 하나 + 나머지 `_구버전`.
4. **끝나면 즉시 `보관`으로** — 진행중을 항상 가볍게.
5. **이름 통일** — `날짜_클라이언트_종류` (예: `250624_코웨이_제안서`).

## 폴더 지도

```
📁 00_자주쓰는        ← 주력 3종 (최상단, 빠른 접근)
   ├── 제안서/
   │   ├── _템플릿/
   │   └── 250624_코웨이_제안서/
   ├── 회사소개/
   │   ├── _최신본          ← 진짜 최신 단 하나
   │   └── _구버전/
   └── 견적/
       ├── _템플릿/
       └── 250624_코웨이_견적/

📁 01_진행중          ← 지금 돌아가는 프로젝트만
   └── 코웨이_안마기/
       ├── 01_브리프·자료
       ├── 02_작업
       ├── 03_납품
       └── _메모

📁 02_보관            ← 끝난 건 전부 (연도별)
   └── 2026/코웨이_안마기/

📁 03_자산            ← 재사용 리소스
   ├── 폰트
   ├── 로고·CI
   ├── 레퍼런스
   └── 포트폴리오 소스
```

## 플랫폼 역할 (단일 기준 원칙)

| 플랫폼 | 단 하나의 역할 | 규칙 |
|--------|---------------|------|
| **노션** | 사령탑(상태·진행) | "지금 뭐가 돌아가는가"의 유일한 기준. 파일은 링크만. |
| **로컬/드라이브** | 파일 실물 | 위 폴더 지도 그대로. |
| **시트** | 견적·리소스 데이터 | 숫자·목록만. 파일은 안 둠. |

- [ ] 회사소개 `_최신본` 단 하나 확정 → 나머지 `_구버전`으로

# (B) 전략적 정리 — 4개 역할의 동시 수행 문제

> 근본 진단(본인 말): "전략 / 실무 / 리더십 / 견적을 **한꺼번에** 한다."
> = 임원급 CD의 전형적 함정. 4개 모자를 동시에 쓰면 레버리지가 0이 되고,
> 가장 가치 있는 일(전략·리더십)이 가장 급한 일(실무·견적)에 자리를 뺏긴다.

## 레버리지 × 에너지 분해

| 역할 | 레버리지 | 에너지 | 대체가능성 | → 처리 |
|------|:---:|:---:|:---:|------|
| **전략** | 최고 | 높음 | 낮음(=당신만) | 🛡️ 보호·확장 |
| **리더십** | 높음 | 높음(확정) | 낮음 | 🛡️ **확장** |
| **실무** | 중 | 최고(취미) | 중 | ✂️ 선별 — POV 걸린 것만 |
| **견적** | 낮음 | 낮음(진빠짐) | 높음 | ⚙️ 시스템화·위임 |

## 핵심 통찰
- **전략·리더십이 둘 다 "레버리지·에너지" 모두 높음** → 흔치 않은 정렬. 망설임 없이 무게 이동.
- **유일한 함정 = 실무.** 가장 좋아해서(취미) 손이 가지만 전략·리더십 시간을 잡아먹음.
  → 프레임 전환: "실무를 줄인다" = "더 좋아하는 일(리더십)로 옮겨간다".
- **에너지(실무)와 레버리지(전략·리더십)가 어긋나 있다** = 동시 진행의 진짜 비용.

## 빼기의 순서 (실행)
1. **견적 → 시스템화.** 템플릿+기준표로 반복/위임. 진 빠지는 정량작업을 매번 새로 안 함. **효과 최대.**
2. **실무 → 선별.** 다 하지 말고 "내 POV가 걸린 핵심"만 직접. 나머지는 팀에. (실무를 넘기는 게 곧 리더십)
3. **번 시간 → 전략·리더십·POV 언어화(첫 마일스톤)에 투입.**

> 연결: 견적 시스템화 + 실무 선별 → 첫 마일스톤("POV 언어화")에 쓸 시간이 바로 여기서 나온다.

# (B-1) 실무 선별 + 컨펌(실무 가이드) 잘하기

> 본인 단서: "실무 가이드도 잘하고 싶다, 컨펌을 많이 하니까."
> → 컨펌은 (B)의 진짜 열쇠. 실무를 '직접'에서 '가이드'로 넘기는 전환점.

## 핵심 통찰 — 컨펌 = 만능 레버
컨펌을 잘하면 한 번에 셋이 풀린다:
- 팀이 기준을 학습 → 알아서 함 → **넘길 실무 증가 → 시간 확보**
- 컨펌 시 "왜 맞고 틀린지" 말하기 = **POV 매일 언어화** (= 첫 마일스톤 자동 진행)
- 그 자체가 **리더십**(에너지 도는 영역)
→ 반대로 컨펌이 부실하면 팀이 못 따라와 결국 본인이 재작업 = 실무를 못 줄임.

## 컨펌 4원칙
1. **기준을 시작(브리프)에 준다** — 컨펌이 길어지는 건 출발점에 기준이 없어서.
2. **"별로야" 대신 "왜"를 말한다** — "[기준]에 안 맞아, 왜냐면 ___" = POV 언어화 + 팀 교육.
3. **방향 컨펌과 디테일 컨펌을 분리** — 방향 OK 후 디테일. (동시에 보면 둘 다 낭비)
4. **3단 신호로 끝낸다** — ✅이대로 / 🔧여기만 / ↩️방향 재검토.

## 실무 선별 기준
판단 한 문장: **"이걸 내가 안 하면 결과물의 격이 떨어지나?"**

| 직접 한다 🖐️ | 넘긴다 ➡️ |
|---|---|
| 내 POV·시그니처가 걸린 핵심 | 반복·정형 작업 |
| 클라이언트 최종 인상 좌우 | 팀이 크는 학습용 작업 |
| 전략적으로 중요한 결정 | 차별점 안 걸린 것 |
