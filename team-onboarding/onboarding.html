팀 전원을 같은 출발선에.
클로드 온보딩 플레이북

클로드를 처음 쓰는 팀원도, Sonnet·Fable 같은 모델을 몰라도 어려움 없이 실무에 투입할 수 있게 한다. 이 문서는 특정 회사 자산에 묶이지 않는다 — 폴더째 복사해 다른 팀·워크스페이스로 그대로 옮길 수 있다.

읽는 시간 15분 공통 온보딩 90분 롤아웃 3단계 목표 습관 통일
접점 3가지 세션 모델 5가지 습관 3단계 롤아웃 그룹·개인 운영 용어 사전 이수 체크리스트
01

어디서 쓰나 — 접점 3가지

같은 클로드지만 들어가는 문이 셋. 문마다 잘하는 일이 다르다.

① 챗 · claude.ai

대화창

아이데이션 · 글쓰기 · 리서치 · 요약 · 시안 피드백. 대부분의 팀원이 여기서 시작한다.

② 프로젝트

맥락 폴더

관련 대화·파일·지침을 한 건으로 묶는 공간. "이 클라이언트 건은 여기서만" — 맥락을 계속 유지.

③ 코드 · Claude Code

작업실

실제 파일 생성/수정·자동화·대량 문서. 필요한 사람만 나중에 배운다.

초기 권장: 대부분은 ①챗 + ②프로젝트만으로 리서치·기획·글쓰기·검토의 80%를 해결한다.

02

세션 = 대화 하나 = 맥락 하나

가장 중요한 개념. 이것만 알면 절반은 끝난다.

새 대화를 열면 클로드는 그 대화 안에서 오간 내용만 기억한다. 옆 대화는 모른다. 사람으로 치면 "이 회의실 안 이야기만 기억하는" 상태다.

세션 관리 3규칙
  1. 주제가 바뀌면 새 세션. A클라이언트 → B클라이언트로 넘어가면 새 대화. 안 그러면 맥락이 섞여 답이 흐려진다.
  2. 세션 이름은 규칙대로. [클라이언트]_[업무]_[날짜] 형식으로 나중에 찾기 쉽게.
  3. 길어진 세션은 결정을 밖으로. 앞부분이 요약되며 디테일이 흐려질 수 있으니, 확정 방향·수치·문구는 그때그때 문서로 남긴다.

컨텍스트 = 클로드가 지금 머릿속에 든 대화 전체(= 화이트보드). 토큰 = 글자 세는 단위. 화이트보드가 다 차면 오래된 걸 요약·삭제한다. → 지금 일에 필요한 것으로만 채운다.

03

모델 — 3줄 규칙이면 끝

이름을 다 외울 필요 없다. 아래 세 줄만.

모델 3줄 규칙
  1. 기본은 Sonnet. 일상 실무 대부분(글·기획·리서치·검토)에 빠르고 충분.
  2. 어려우면 Opus. 복잡한 추론·긴 전략 문서·까다로운 판단.
  3. 단순·급하면 Haiku, 최고난도 장기전이면 Fable.
모델비유언제
Haiku
4.5
인턴 (빠른 손)단순·반복·빠른 처리 — 분류, 짧은 요약, 초안 대량 생성
Sonnet
기본값 · 5
실무 담당일상 실무 전반 — 기획·글쓰기·리서치·시안 피드백·코딩
Opus
4.8
시니어복잡한 전략·긴 제안서·장시간 자동 작업·어려운 판단
Fable
5 · 비쌈
특급 전문가최고 난도·초장기 작업. 꼭 필요할 때만

실전: 헷갈리면 Sonnet으로 시작 → 답이 얕으면 "이거 Opus로 다시 해줘"로 올린다. 위로 올리는 건 쉽다. (모델 이름은 갱신될 수 있다. 새 이름이 보이면 상위 버전으로 이해하고 3줄 규칙 그대로 적용.)

04

잘 부리는 5가지 습관

전원 공통. 완벽한 프롬프트 엔지니어가 될 필요는 없다.

  • 01맥락을 먼저 준다. "누구를 위한, 무슨 목적의 일인지"를 앞에 깔면 결과가 확 좋아진다.
  • 02대화로 다듬는다. "더 짧게 · 이 톤으로 · 3안만 남기고" — 왕복이 정상이다.
  • 03예시를 준다. 원하는 톤/형식의 샘플 하나를 붙이면 적중률이 오른다.
  • 04주제가 바뀌면 새 세션.
  • 05사실·숫자·PII·톤은 사람이 검수 후 발송. 특히 외부 발송물.
나쁨배너 문구 뽑아줘
좋음20대 여성 화장품 여름세일 배너. 톡톡 튀되 과하지 않게, 15자 이내 5개.
05

3단계 롤아웃

한 번의 교육이 아니라, 베이스라인 통일 → 실무 적용 → 팀 표준화의 순서.

1
1주차 · 베이스라인 통일

전원이 개념을 이해하고 혼자 시작할 수 있다

기초개념 각자 읽기 → 90분 공통 온보딩 → 치트시트 핀 고정 → 이수 체크리스트 8항목 통과.

→ 전원 체크리스트 통과 = "같은 출발선" 달성
2
2–4주차 · 실무 적용

각자 자기 업무에 클로드를 붙인다

역할별 레시피 3개를 실제 업무에 적용 → 주 1회 15분 "이번 주 뭘 아꼈나" 공유 → 잘 된 프롬프트는 공유 저장소에 적립.

→ 1인당 검증된 워크플로 ≥ 3개
3
2개월차~ · 팀 표준화

개인 노하우를 팀 자산으로 전환

반복 프롬프트 → 공유 템플릿/프로젝트로 승격 → 자주 쓰는 검수·산정 흐름은 에이전트/스킬로 패키징(선택).

→ "개인이 잘 쓰는 팀" → "팀 시스템으로 잘 쓰는 팀"
06

그룹 사용 vs 개인 사용

무엇을 각자 하고, 무엇을 팀 공유로 관리하나.

개인 영역 · 속도
  • 각자의 실무 대화·초안·실험
  • 개인 프롬프트, 일회성 요청
  • 자유롭게 빠르게 시도
그룹 영역 · 일관성
  • 팀 공통 규칙·검증된 프롬프트
  • 클라이언트 프로젝트 노트·템플릿
  • 노하우 축적 · 중복 제거

원칙: 개인이 잘 쓴 게 반복되면 → 그룹으로 승격한다. 계정은 개인별(보안·책임 추적), 기밀·PII는 입력 전 마스킹.

세션 이름[클라이언트]_[업무]_[날짜]예: 코웨이_상세페이지_0720
프로젝트 이름YYYYMM_클라이언트_업무예: 202607_코웨이_상세페이지
07

용어 사전 — "이 단어가 대체 뭐야?"

스킬·마크다운·깃허브가 뭔지 몰라도 일은 시작된다. 이 사전은 겁먹지 말라고 있는 것.

먼저 안심하세요. 당장 알아야 할 건 거의 없습니다. 챗에서 한국어로 부탁하는 것만으로 대부분의 일이 됩니다. 아래는 "우리 작업 환경에서 지나가다 보게 될 간판"일 뿐 — 뜻만 알면 안 무섭습니다. ALWAYS = 알면 편함 / SKIP = 지금은 신경 끄기.
A · 파일·문서 형식
마크다운 (.md)알면 편함

서식 있는 글을 아주 간단한 기호로 쓰는 가벼운 문서 형식. 이 온보딩 문서들이 전부 마크다운이다.비유 — 제목·굵게·목록이 되는 초경량 메모장 (워드의 가벼운 버전)

HTML몰라도 됨

웹페이지를 만드는 형식. 결과물을 "보기 좋게 전달"할 때 클로드가 대신 만들어준다. 이 페이지가 그 예.

B · 깃허브 생태계 — 팀 공동 창고
대부분의 팀원은 깃허브를 직접 안 만져도 됩니다. "우리 결과물이 여기 보관·버전관리되고 링크로 공유된다" 정도만 알면 충분.
깃허브 (GitHub)알면 편함

문서·코드를 팀이 함께 보관하고 누가 언제 뭘 바꿨는지 전부 기록하는 클라우드 창고.비유 — 구글 드라이브 + 강력한 "변경 이력 되감기"

저장소 · 커밋 · 브랜치 · PR원리는 몰라도 됨

저장소 = 프로젝트 폴더 전체 · 커밋 = "저장+변경 메모" 스냅샷(게임 세이브) · 브랜치 = 원본 안 건드리는 작업 사본 · PR = "이 수정본 합쳐도 될까요?" 결재 상신.

C · 클로드 확장 개념
스킬 (Skill)알면 편함

특정 작업을 "이렇게 해라"라고 미리 패키징해 둔 설명서 묶음. 클로드가 필요할 때 알아서 꺼내 쓴다. 매번 같은 지시를 반복 안 해도 됨.비유 — 자주 쓰는 요리의 레시피 카드

에이전트 (Agent)알면 편함

특정 역할에 특화된 클로드 분신. "견적 담당", "트렌드 리서치 담당"처럼 부르면 그 관점으로 일한다.스킬 = 작업 방법 설명서 / 에이전트 = 그 방법을 든 담당자

아티팩트 (Artifact)알면 편함

클로드가 만든 결과물(웹페이지·문서 등)을 링크로 바로 볼 수 있게 발행한 것. 이 온보딩 페이지가 그 예.

D · 연결·도구
MCP · 커넥터 (Connector)알면 편함

클로드를 외부 도구(노션·지메일·구글드라이브·피그마 등)에 연결하는 다리. 연결하면 그 안의 자료를 읽거나 쓸 수 있다. (연결은 보통 관리자가 세팅)주의 — 연결된 자료는 기밀·개인정보일 수 있으니 보안 규칙 준수

Claude Code · 터미널(CLI)몰라도 됨

검은 화면에 명령으로 파일·자동화를 직접 다루는 개발자용 환경. 대부분의 팀원은 지금 몰라도 된다. 파일 대량 생성·자동화가 필요해질 때 배우면 됨.

핵심: 위 단어가 대화나 문서에 나와도 당황하지 마세요. "아 그거 그런 뜻이었지" 하고 넘어가면 됩니다. 정말 필요해지는 순간, 그때 이 사전을 다시 펴면 됩니다.

08

이수 체크리스트

전원 8/8 통과 = "같은 출발선" 달성. 기준은 완벽이 아니라 "혼자 겁 없이 시도할 수 있다".

  • 챗·프로젝트·코드 3접점의 차이를 한 문장씩 설명할 수 있다
  • "세션 = 맥락"을 이해하고, 주제가 바뀌면 새 세션을 연다
  • 모델 3줄 규칙을 안다 (기본 Sonnet / 어려우면 Opus / 양 끝은 Haiku·Fable)
  • 나쁜 요청을 "맥락 먼저 + 구체적으로" 좋은 요청으로 고칠 수 있다
  • 결과를 대화로 다듬어 본다 (더 짧게 / 톤 변경 / N안 남기기)
  • 클라이언트/건별로 프로젝트를 하나 만들어 봤다
  • 사실·숫자·PII·톤은 사람이 검수 후 발송한다는 원칙을 안다
  • 막혔을 때 어디에 물어보는지 안다 (팀 공유 채널)
