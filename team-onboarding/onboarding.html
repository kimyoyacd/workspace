01 · Session & Models

세션 = 대화 하나 = 기억 하나

이것만 알면 절반은 끝나요. 새 대화를 열면 클로드는 그 대화에서 나눈 얘기만 기억해요 — 옆 대화는 몰라요. 이 회의실에서 한 얘기만 기억하는 사람인 셈이에요.

핵심 개념
세션 쓰는 3가지 규칙
  1. 주제가 바뀌면 새 대화. 고객 A 얘기하다 고객 B로 넘어가면 새 대화를 열어요. 안 그러면 얘기가 섞여서 답이 흐려져요.
  2. 대화 이름을 규칙대로. [고객]_[일]_[날짜] — 예: 코웨이_상세페이지_0720. 나중에 찾기 쉬워요.
  3. 대화가 길면 중요한 건 밖에 적어둬요. 길어지면 앞부분이 요약돼 세세한 게 흐려질 수 있어요. 정한 방향·숫자·문구는 그때그때 문서로.

컨텍스트 = 클로드가 지금 들고 있는 대화 전부(칠판). 토큰 = 그 글자를 세는 단위. 칠판이 꽉 차면 오래된 걸 지워요 → 지금 일에 필요한 얘기로만 채워요.

모델 — 딱 3줄만 기억해요

이름을 다 외울 필요 없어요

  1. 보통은 Sonnet. 매일 하는 일(글·기획·자료찾기·검토)엔 이게 빠르고 충분해요.
  2. 어려우면 Opus. 복잡한 생각, 긴 제안서, 까다로운 판단일 때 올려요.
  3. 빠르고 간단하면 Haiku, 아주 어렵고 오래 걸리면 Fable.
모델비유언제 써요
Haiku인턴 (손이 빠름)간단·반복·빠르게 — 분류, 짧은 요약, 초안 여러 개
Sonnet · 기본실무 담당매일 하는 일 거의 다 — 기획·글쓰기·자료찾기·시안 봐주기
Opus선배복잡한 전략 · 긴 제안서 · 오래 걸리는 자동 작업 · 어려운 판단
Fable · 비쌈특급 전문가아주 어렵고 오래 걸리는 일. 꼭 필요할 때만

이렇게 해요: 헷갈리면 Sonnet으로 시작하고, 답이 얕으면 "이거 Opus로 다시 해줘" 하면 돼요. 이름이 바뀌어도 이 3줄은 그대로예요.

02 · Glossary

명칭 사전 — 이름 뒤의 진짜 정체

스킬·에이전트·커밋… 이름이 거창해서 어렵게 느껴지지만, 정체를 알면 하나도 안 무서워요. 각 항목은 "무엇이 아닌지"부터 말해요.

핵심알면 편함몰라도 됨
미리 결론: 여기 나오는 것 대부분의 정체는 폴더 속 문서 한 장이에요. 파일이 있으면 기능이 생기고, 지우면 사라져요. 마법도, 프로그램 설치도 아니에요.
스킬Skill알면 편함

스킬은 마법이 아닙니다. 클로드가 적절한 순간에 꺼내 읽는 미리 작성된 지침서예요.

"덱을 만들 땐 이 색·이 여백·이 규격으로" 같은 방법을 한 번 적어두면, 관련 요청이 올 때 클로드가 알아서 펼쳐 봐요. 스킬 덕분에 매번 즉흥적으로 하는 대신 일정한 기준을 유지할 수 있어요.

.claude/skills/스킬명/SKILL.md ← 이 파일 하나가 스킬의 전부
에이전트Agent알면 편함

에이전트는 또 다른 AI가 아닙니다. 역할 지시서를 입은 같은 클로드예요.

"너는 시장조사 담당. 출처 없는 숫자는 쓰지 마" 같은 역할 문서를 만들어두면, 부를 때마다 그 시선으로만 일해요. 견적 볼 때와 트렌드 볼 때 기준이 섞이지 않는 이유예요. 스킬이 "하는 방법"이라면, 에이전트는 "그 방법을 든 담당자"예요.

.claude/agents/에이전트명.md ← 담당자 한 명 = 파일 한 장
CLAUDE.md클로드 엠디핵심

프로그램이 아닙니다. 클로드가 일을 시작할 때마다 맨 먼저 읽는 팀 규칙 메모예요.

"우리 팀 말투는 이렇고, 이 폴더는 건드리지 마" 같은 걸 한 번 적어두면 매번 다시 설명할 필요가 없어요. 팀의 약속을 이 파일에 쌓아갈수록 클로드가 점점 "우리 팀 사람"처럼 일해요.

저장소 루트의 CLAUDE.md ← 팀 규칙의 집
프로젝트Project핵심

특별한 기능이 아닙니다. 대화·파일을 한 가지 일 단위로 모아두는 폴더예요.

"이 고객 일은 여기서만" 하고 정해두면, 그 안의 대화들이 같은 맥락을 공유해요. 기억이 이어지는 범위를 정해주는 울타리라고 생각하면 돼요. 일이 끝나면 닫아요.

아티팩트Artifact알면 편함

별도의 앱이 아닙니다. 클로드가 만든 결과물을 링크 하나로 볼 수 있게 내보낸 것이에요.

웹페이지·문서·표를 만들면 클로드가 링크로 발행해줘요. 팀원은 설치 없이 링크만 열면 돼요. 지금 보고 있는 이 페이지가 바로 그 예예요.

MCP · 커넥터Connector알면 편함

해킹 같은 게 아닙니다. 클로드와 노션·지메일·피그마 사이에 놓는 다리예요.

다리를 놓은 만큼만 그 안의 자료를 읽고 쓸 수 있어요. "어제 온 메일 요약해줘"가 되는 건 지메일에 다리가 놓여 있기 때문이에요. 안 쓸 땐 꺼둬도 돼요.

주의 — 다리 건너편엔 비밀·개인정보가 있을 수 있어요. 연결은 팀 보안 규칙을 따라요.
마크다운.md알면 편함

코드가 아닙니다. # 몇 개로 제목·목록이 되는 가벼운 문서예요.

메모장과 워드 사이 어디쯤이에요. 스킬도, 에이전트도, CLAUDE.md도 결국 전부 이 문서 한 장이에요 — 그래서 "파일이 곧 기능"이라는 말이 성립해요.

세션 · 컨텍스트 · 토큰Session · Context · Token핵심

세션은 저장 공간이 아닙니다. 회의실 하나의 기억이에요.

컨텍스트는 그 회의실의 칠판(지금 들고 있는 대화 전부), 토큰은 칠판의 글자를 세는 단위예요. 칠판이 꽉 차면 오래된 게 지워져요 — 그래서 "주제가 바뀌면 새 대화"가 규칙이에요.

모델Sonnet · Opus · Haiku · Fable3줄이면 충분

서로 다른 회사 제품이 아닙니다. 같은 클로드의 체급 차이예요.

인턴(Haiku) — 실무 담당(Sonnet) — 선배(Opus) — 특급 전문가(Fable). 기본은 Sonnet, 어려우면 Opus. 이름이 바뀌어도 이 구조는 그대로예요. (01에 표 있어요)

  2. 거꾸로 물어보기 — 뭘 써야 할지 모르겠으면 "좋은 결과를 내려면 나한테 뭘 물어봐야 해?" 하고 클로드에게 먼저 질문시키세요.

비유 — 구글 드라이브 + 아주 강력한 되돌리기

주요 명칭 — 흐름으로 보면 한 번에 이해돼요

순서대로 일어나는 일이에요

1add담기바꾼 파일을 장바구니에 담아요
2commit저장 + 메모"뭘 바꿨는지" 메모와 함께 세이브
3push창고에 올리기내 컴퓨터 → GitHub 창고로
4Pull Request합쳐도 돼요?"내 작업을 원본에 합쳐도 될까요?" 확인 요청
5Merge합치기OK되면 원본(main)에 합쳐져요

Branch(브랜치)는 특정 시점에서 갈라진 작업용 복사본이에요. 원본(main)을 안 건드리고 복사본에서 작업한 뒤, 위의 4→5(PR→Merge)로 합쳐요. 그래서 실수해도 원본은 안전해요.

이렇게"수정 내용 Push하고 main 브랜치로 Pull Request 해 줘"창고에 올리고 "합쳐도 돼요?" 요청까지 한 번에
한 수 위"작업이 끝날 때마다 add, commit 해 줘"를 CLAUDE.md에 적어두기그러면 매번 말할 필요도 없어요 — 규칙이 되니까요 (02 명칭 사전의 CLAUDE.md 참고)

새 세션 열 때 체크 — 깃허브가 연결돼 있나요?

이게 안 돼 있으면 작업해도 팀 창고에 안 남아요

Code(claude.ai/code)에서 새 대화를 열면, 입력창 근처에 저장소·브랜치 칩이 보여야 해요. 이 칩이 우리 팀 저장소로 연결돼 있어야 클로드가 만든 파일이 깃허브에 저장되고 히스토리가 남아요. 칩이 없으면 → 그 세션의 작업은 팀 기록에 남지 않아요.
이 상태로 시작하는 게 정답이에요

☁ 우리팀-저장소 </> 우리팀-저장소 ⎇ main

  1. 새 대화를 열면 입력창 위/아래의 칩부터 확인해요 — 저장소 이름 + 브랜치가 보이면 연결된 거예요.
  2. 칩이 없으면 + 버튼(소스 추가)으로 팀 저장소를 선택해요. 처음 한 번은 GitHub 연결 승인(로그인)이 필요해요.
  3. 브랜치는 main 기준으로 시작 — 실제 작업은 클로드가 알아서 새 Branch를 만들어 진행하고, 끝나면 PR로 합쳐요(위 흐름 그대로).

기억할 것: 히스토리가 필요한 프로젝트성 작업(리서치 보고서·프로젝트 노트·팀 규칙)은 반드시 이렇게 저장소가 연결된 Code 세션에서 해요. 챗에서만 하면 대화가 끝나는 순간 기록이 사라져요.

클로드에게 일을 시킬 때 딱 한 가지만 기억하세요 — 누구를 위한, 무슨 일인지부터 말한다.

"주제가 바뀌면 새 대화" 습관 만들기

클로드는 지금 열린 대화 하나만 기억해요(옆 대화는 몰라요). 그래서 고객 A 얘기하다 고객 B로 넘어가면 새 대화를 열어야 답이 안 섞여요.

실습 ②: 지금 진행 중인 일 하나로 프로젝트(폴더)를 만들고, 관련 대화를 그 안에서만 해보기. "이 고객 일은 여기서만" — 이러면 맥락이 이어져요.

06 · Team Ops

팀으로 쓰기 vs 혼자 쓰기

뭘 각자 하고, 뭘 같이 관리할까요. 원칙은 하나 — 혼자 잘 쓴 게 자꾸 반복되면 팀 것으로 올려요.

혼자 · 빠르게

각자 일 대화·초안·실험 · 개인 요청, 한 번 쓰고 마는 것 · 자유롭게 빨리 해보기

팀 · 똑같이 맞추기

팀 공통 규칙 · 잘 된 요청 모음 · 고객 프로젝트 메모·틀 · 노하우 쌓기 · 겹치는 일 줄이기

계정은 각자 쓰고(보안), 비밀·개인정보는 넣기 전에 가려요.

07 · Checklist & FAQ

다 됐는지 확인 목록

8개 다 통과하면 "같은 출발선" 완성. 기준은 완벽이 아니라 "혼자서도 겁 없이 해본다"예요.

8개 통과 = 완료
  • 챗·프로젝트·코드, 세 가지 쓰는 곳의 차이를 한 문장씩 말할 수 있어요
  • "세션 = 기억 하나"를 알고, 주제가 바뀌면 새 대화를 열어요
  • 모델 3줄을 알아요 (보통 Sonnet / 어려우면 Opus / 양 끝은 Haiku·Fable)
  • 대충 한 요청을 "상황부터 + 자세히"로 고칠 수 있어요
  • 답을 대화로 다듬어 봐요 (더 짧게 / 느낌 바꾸기 / 몇 개만 남기기)
  • 고객·일별로 프로젝트를 하나 만들어 봤어요
  • 사실·숫자·개인정보·말투는 사람이 확인하고 보낸다는 걸 알아요
  • 막히면 어디에 물어보는지 알아요 (팀 공유 채널)

자주 묻는 질문

여기 없으면 팀 공유 채널에

Q. 대화가 자꾸 흐려져요

언제 새 대화를 열어요?

주제가 다르면 바로 새 대화. 고객 A → 고객 B, 배너 → 기획으로 넘어가면 새로 열어요.

Q. Sonnet이 부족해요

Opus로 바꿔도 되나요?

네. "답이 얕으면 Opus로 다시 해줘" 하면 바로. 올리는 것도 내리는 것도 자유예요.

Q. 고객 정보 입력해도 돼요?

고객 이름·가격·계약서는?

발송 전 사람이 필수 확인. 구체적 수치·이름은 최소화, "외부 반출 금지" 자료는 넣지 않아요. 마스킹해서 입력해요.

Q. 프로젝트는 언제 닫아요?

계속 열어두면 안 되나요?

고객 일이 끝나면 닫아요. "한 가지 일" 단위니까요. 같은 고객이 다시 오면 새로 만들면 돼요.

Q. 한 달 전 대화를 다시

이어서 할 수 있나요?

새 대화로 시작하되 핵심만 붙여넣기. 이전 대화는 읽을 수 있지만, 이으려면 핵심을 옮겨 붙여요.

Q. 계속 틀린 답을 줘요

클로드가 이상한 건가요?

요청이 불명확했을 수 있어요. ① 예시 붙이기 ② 상황 자세히 ③ Opus로 올리기 — 보통 이 중 하나로 해결돼요.

Q. 내 요청이 반복돼요

자동화할 수 있나요?

네. 같은 요청을 월 2회 이상 하면 팀 채널에 말해주세요. 스킬/에이전트로 만들 수 있어요.

08 · Master Prompt

마스터 프롬프트 — 나만의 완전한 세팅

7일을 완주했다면, 아래를 복사해 빈칸을 채우고 클로드에 붙여넣으세요. 나만의 세팅 계획을 통째로 받을 수 있어요.

Copy & Fill

빈칸 4곳만 채우면 돼요. 결과로 프로젝트 구성 → 스킬 → 커넥터 → 예약 작업 → CLAUDE.md까지 한 번에 나와요.

내 업무에 맞는 제대로 된 Claude 세팅을 만들고 싶어.

나에 대해: [3~4줄로 나를 소개]
내 주요 작업: [Chat / Code에서 각각 뭘 하는지]
주간 반복 업무: [1, 2, 3]
쓰는 도구: [Gmail, Drive, Notion 등]

완전한 세팅 계획을 만들어줘:
① 어떤 프로젝트를 만들고 각각 어떤 지침이 필요한지
② 먼저 만들 스킬
③ 설치할 플러그인
④ 우선할 MCP 커넥터
⑤ 예약 작업 아이디어
⑥ CLAUDE.md 템플릿
