# AI 원스톱 제작 체계 제안 — "키워드만 주시면, 영상까지 한 번에"

> 2026-07-10 넷마블 회의에서 확인된 문제에 대한 해결 방향 계획
> 관련: 만족도 조사 AI 3.72 (집중 개선), Figma 발표 덱 08번 슬라이드

## 1. 오늘 회의에서 확인한 문제

| 넷마블이 원하는 것 | 지금 걸리는 것 |
|---|---|
| 기획 아이디어·키워드만 넣으면 영상까지 한 번에 | 기획 TO / 이미지 TO / 영상 TO로 창구가 나뉘어 있어, 어디에 뭘 요청해야 할지 복잡함 |
| 빠르고 가벼운 진행 | 단계마다 검수가 반복되어 피로함 |
| — | 결과적으로 **AI 업무를 하이브랩과 잘 안 하게 되는 구조** |

핵심: AI 물량이 안 들어오는 건 능력 문제가 아니라 **구조(창구·검수) 문제**. 넷마블의 불편을 우리가 흡수하는 쪽으로 구조를 바꾸면 AI 협업 물량 자체가 늘어난다.

## 2. 해결 방향 — 세 가지 축

### A. 창구를 하나로 — "AI 패스트트랙"
- 요청 폼은 딱 5칸: **게임 / 키워드(아이디어) / 용도·매체 / 사이즈 / 참고자료(선택)**
- 어느 TO로 갈지, 기획→이미지→영상 배분은 **하이브랩 내부에서 알아서 처리** — 넷마블은 한 번만 요청
- 전담 담당자 1명이 접수부터 전달까지 진행 상황을 안내 (문의 창구도 한 명)

### B. 검수를 "고르기"로 — 기본 2터치
- **1터치(선택)**: 시작할 때 방향 확인 — 원하면 생략 가능
- **2터치**: 완성본을 A/B/C 버전으로 드리고 **고르기 + 수정 요청**
- 중간 검수 없음. AI는 다시 뽑는 비용이 낮아서, 중간에 물어보는 것보다 **일단 만들어서 보여드리고 고르게 하는 것**이 더 빠름
- "검수해 주세요"(숙제) → "골라 주세요"(선택)로 경험 자체를 바꿈

### C. 작게 시작 — 4주 파일럿
- 타이틀 1개 선정, 주 N건, **접수 후 48~72시간 안에 러프 영상 전달**
- 4주 후 결과 보고: 요청→전달 시간 / 검수 횟수 / 실제 사용률
- 숫자로 보여준 뒤 타이틀·물량 확대 여부를 함께 결정

## 3. 흐름 비교

**AS-IS** (창구 3개 · 검수 3회+)
```
기획 TO 요청 → 기획 검수 → 이미지 TO 요청 → 시안 검수 → 영상 TO 요청 → 영상 검수 → 완성
```

**TO-BE** (창구 1개 · 검수 1~2회)
```
패스트트랙 접수(폼 5칸) → [방향 확인 1회, 생략 가능] → 러프 영상 A/B/C 전달(48~72h) → 고르기·수정 → 완성
```

## 4. 넷마블과 미리 정할 것

1. **AI 사용 허용 범위** — IP 제한 타이틀(킹스로드 등)은 패스트트랙 대상에서 제외하거나 비캐릭터 요소만 적용
2. **러프 퀄리티 기대치** — 첫 전달물은 방향 확인용 러프임을 합의 (완성 퀄리티는 선택 후 마감 단계에서)
3. **파일럿 조건** — 대상 타이틀, 주당 건수, 기간
4. **선택 방식의 책임 범위** — 방향 확인을 생략하는 경우, 버전 선택까지가 넷마블 몫이고 이후 수정은 우리 몫이라는 선 긋기

## 5. 내부 준비 사항

- [ ] 접수 폼(5칸) 제작 및 접수 채널 결정 (시트/메신저/노션 등)
- [ ] 내부 라우팅 규칙: 접수 → 기획/이미지/영상 배분 기준, 전담 담당자 지정
- [ ] 기획→이미지→영상 AI 파이프라인 표준화 (SB 자동화, 사이즈 변환 포함)
- [ ] 48~72시간 SLA가 가능한 물량 상한 산정
- [ ] TO 정산 구조와 통합 요청의 매핑 정리 — 기획/이미지/영상 TO를 어떻게 차감할지 내부 기준 필요 (⚠️ 계약 이슈, 영업과 협의)
- [ ] 파일럿 성과 지표 기록 방식 (요청→전달 시간, 검수 횟수, 사용률)

## 6. 커뮤니케이션 포인트

- "AI를 더 써 달라"가 아니라 **"AI를 쓰기 편하게 우리가 바꾸겠다"** 로 접근
- 검수를 없애는 것이 아니라 **선택으로 바꾸는 것** — 품질 통제권은 여전히 넷마블에 있음
- 파일럿은 넷마블 입장에서 리스크 없음: 기존 TO 구조는 그대로 두고, 패스트트랙을 옆에 하나 더 여는 방식
