# 02. GitHub 가이드 — 디자이너를 위한 겁먹지 않는 협업법

> "GitHub? 개발자만 쓰는 거 아냐?" — 아닙니다. 우리한테 GitHub은 **팀 공용 창고 + 자동 되돌리기(Undo) 무한** 기능입니다. 이 문서만 따라 하면 됩니다.

## 왜 쓰나요?

- **한 벌의 규칙·자산을 팀 전원이 똑같이** 봅니다. (각자 다른 버전으로 안 헤맴)
- 누가 언제 뭘 바꿔도 **전부 기록·복구 가능**. 실수해도 되돌릴 수 있음.
- 세션은 임시 공간이라 껐다 켜면 사라집니다. **GitHub에 저장(push)해야 남습니다.**

## 5개 단어만 알면 됩니다

| 단어 | 쉬운 뜻 | 디자인 비유 |
|---|---|---|
| **저장소(repo)** | 프로젝트 폴더 전체 | 회사 공유 드라이브 |
| **브랜치(branch)** | 작업용 사본 | "저장하기 전" 임시 레이어 |
| **커밋(commit)** | 저장 + 메모 | PSD 버전 저장 (v1, v2…) |
| **푸시(push)** | 내 저장을 창고로 올림 | 공유 드라이브에 업로드 |
| **PR (Pull Request)** | "이거 합쳐도 될까요?" 요청 | 실장 컨펌 요청 |

## 우리 팀 작업 흐름 (표준)

**핵심 원칙: `main`(본선)에 직접 손대지 않는다. 항상 브랜치에서 작업 → PR로 컨펌 → 합친다.**

```
main (본선, 항상 안전)
  └─ claude/코웨이-노트-추가   ← 여기서 작업하고
         └─ PR 올림 → 실장 확인 → main에 합침(merge)
```

### 방법 A) Claude Code로 (권장 — 제일 쉬움)

명령어 외울 필요 없이 **말로** 시키면 됩니다:

1. *"이 작업 새 브랜치 만들어서 진행해줘"* → Claude가 브랜치 생성
2. 작업 끝나면 *"커밋하고 푸시해줘"* → Claude가 저장·업로드
3. *"PR 올려줘"* → Claude가 초안(draft) PR 생성 → 실장이 웹에서 확인·머지

> Claude Code는 기본적으로 **작업 전용 브랜치에서만** 일하고 `main`에 직접 안 올립니다. 안전.

### 방법 B) GitHub 웹사이트로 (문서 간단 수정)

1. github.com에서 저장소 열기 → 고칠 파일 클릭
2. 연필(✏️) 아이콘 → 수정
3. 아래 **"Commit changes"** → *"Create a new branch... and start a pull request"* 선택
4. PR 생성 → 실장 확인 후 머지

## 커밋 메시지, 이렇게 쓰세요

무엇을 왜 바꿨는지 한 줄로. 한국어 OK.

- ✅ `코웨이 리뉴얼 프로젝트 노트 추가`
- ✅ `견적 단가표 2026 상반기 기준 업데이트`
- ❌ `수정`, `ㅁㄴㅇㄹ`, `test` (나중에 못 알아봄)

## 자주 겪는 상황 Q&A

**Q. 실수로 잘못 저장했어요.**
A. 괜찮습니다. GitHub은 모든 버전이 남아 되돌리기 가능. Claude에게 *"방금 커밋 되돌려줘"* 라고 하세요.

**Q. `main`에 바로 올려도 되나요?**
A. ❌ 안 됩니다. 항상 브랜치 → PR → 실장 컨펌. 이게 사고를 막습니다.

**Q. 충돌(conflict)이 났대요.**
A. 같은 파일을 두 명이 동시에 고쳤을 때 생깁니다. 당황 말고 Claude에게 *"충돌 정리해줘"* 하거나 실장에게 알리세요.

**Q. PR은 누가 머지하나요?**
A. **실장(또는 지정 리뷰어)**. 팀원은 PR 올리는 데까지, 최종 합치기는 컨펌 담당이 합니다.

## 절대 GitHub에 올리면 안 되는 것

- 🔒 비밀번호·API 키·토큰·접속 정보
- 🔒 클라이언트 내부 비공개 자료, 계약서 원본, 개인정보(PII)
- 🔒 최종 견적 협상가 등 민감 숫자 (필요 시 별도 관리)

> 헷갈리면 올리기 전에 실장에게 확인. 한번 올라간 건 삭제해도 기록에 남을 수 있습니다.

## 요약 한 줄

**브랜치에서 작업 → 커밋(메모 남겨 저장) → 푸시 → PR로 컨펌 요청.** 막히면 Claude에게 말로 시키면 됩니다.
